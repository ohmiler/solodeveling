# Solodeveling 0.3.0

Solodeveling 0.3.0 is an alpha feature release that reduces workflow handoffs for
clear Standard changes and makes backend delivery more proportional without lowering
Critical security, recovery, or release boundaries.

## Install and operate

Node.js users can install into the current project without installing Python:

~~~console
npx solodeveling install
~~~

Persistent Node and isolated Python options expose the same command:

~~~console
npm install -g solodeveling
uv tool install solodeveling
pipx install solodeveling
~~~

Run solodeveling check after installation or upgrade. Use
solodeveling uninstall to remove only unchanged managed files.

## Changes since 0.2.0

- Added solodeveling-standard-delivery, which owns shape, plan, execution, and
  verification for clear authorized non-Critical Standard work in one uninterrupted
  workflow.
- Checkpointed Standard lifecycle persistence so one active WORK/EVIDENCE pair is
  opened, updated only at meaningful boundaries, and archived once.
- Reused current verification evidence when its scope still covers the change,
  limited manual browser review to unproven visual or interaction quality, and kept
  transient browser artifacts out of durable project memory by default.
- Added bounded verification triage before routing reproducible or unexplained
  failures to debugging.
- Made backend API and query work Standard by default with a fail-closed Quick
  carve-out only for contract-stable, read-only, non-sensitive, no-effect changes
  proven by a focused contract test.
- Added one shared backend boundary record, effect-specific query, mutation, webhook,
  and migration gates, provider-capability triage, and a proportional additive
  migration template.
- Added a preregistered Solodeveling 0.2.0-versus-no-skill benchmark harness. Its
  offline fixtures pass, but the 30-call study has not run and supplies no comparative
  speed or quality result.

## Security and compatibility

The public CLI, project-memory schema version 1, installation behavior, and supported
Python and Node versions remain compatible with 0.2.0. Routing behavior intentionally
changes, so agents should reload the installed 0.3.0 skills before evaluating it.

Credential, session, authorization-decision, payment-fulfillment, sensitive-data,
destructive migration, production, secret, cryptographic, and safety-critical
changes remain Critical. The combined Standard workflow must not bypass specialized
security, release, diagnosis-only, or verification-only routing.

The complete cross-ecosystem release set remains bound to one exact source revision:
two Python distributions, six native executables, one npm tarball, one CycloneDX
SBOM, release notes, SHA-256 checksums, a release-set manifest, and GitHub
attestations.

## Accepted local-candidate gap

Three release-routing scenarios were preregistered for clear Standard frontend,
clear Standard backend, and Critical backend readiness behavior. Their offline
schema, harness, and dry-run checks pass. Tenant policy prevented the external Codex
calls even after informed owner approval, so no live routing result exists.

On 2026-07-22 the owner explicitly accepted this missing evidence only to build and
inspect a local 0.3.0 candidate. The acceptance does not prove the scenarios pass and
does not authorize a tag, GitHub Release, registry publication, or public readiness
claim. Resolve the live gate in an approved environment or obtain a new explicit
release-level risk acceptance before any public release decision.

## Known limitations

Tier 1 remains unverified. The combined Standard and backend routing changes have
deterministic scenario coverage but no fresh live release-pilot result. Native
executables are not platform code-signed. Solodeveling still makes no controlled
cross-framework speed or quality superiority claim.
