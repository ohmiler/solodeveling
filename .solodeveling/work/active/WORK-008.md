id: WORK-008
title: Harden the 0.1.0 release candidate and publication gate
status: active
level: critical
type: release
goal: Produce a source-bound, reviewable Solodeveling 0.1.0 release candidate with release notes, a validated CycloneDX SBOM, verifiable artifact provenance instructions, and a publication decision boundary that cannot publish without separate user authority.
scope: Candidate identity and source revision, wheel and source distribution, deterministic manifest and checksums, release notes, CycloneDX SBOM generation and validation, GitHub artifact-attestation preparation, PyPI Trusted Publishing prerequisites, least-privilege workflow design, recovery, documentation, tests, and evidence.
out_of_scope: Creating or pushing a version tag, creating a GitHub Release, configuring a PyPI project or GitHub environment, requesting OIDC tokens, generating a repository attestation, uploading to PyPI, deleting branches, changing repository visibility, claiming Tier 1 runtime support, or spending paid agent-evaluation budget.
acceptance:
- The release candidate identity binds version 0.1.0, an exact source revision, wheel and source-distribution digests, SBOM digest, build inputs, and intended target without relying on mutable branch names.
- The built candidate contains one wheel and one source distribution plus verified checksums, canonical inventory, release notes, and a machine-readable CycloneDX JSON SBOM generated from the isolated installed distribution and its resolved runtime dependencies.
- Automated tests reject missing, malformed, mismatched, duplicate, symlinked, stale-version, or unbound candidate artifacts and validate the SBOM with an official CycloneDX implementation.
- Release notes accurately describe supported installation paths, security controls, compatibility, evidence tier, limitations, upgrade and uninstall guidance, and known gaps without claiming unverified Tier 1 behavior.
- A manual, non-publishing GitHub candidate workflow is designed with full-SHA action pins and minimum permissions; provenance attestation requires an explicit authorized invocation and is never available to untrusted pull-request code.
- PyPI Trusted Publishing documentation names the exact repository, workflow, environment, package, authorization boundary, and recovery steps while recording that no GitHub environment or PyPI project exists yet.
- Full regression, ten official skill validations, protocol validation, isolated candidate build/install, SBOM validation, dependency audit, cross-platform CI, compilation, dependency health, and diff checks pass or are recorded as explicit gaps.
- No tag, GitHub Release, attestation, registry upload, environment configuration, or publication occurs during WORK-008.
risks:
- An SBOM generated from the wrong environment can include build tools or omit runtime dependencies and create false confidence.
- A provenance workflow with broad permissions or pull-request reachability can expose an identity token to untrusted code.
- Package-name availability is not a reservation and can change before publication.
- Candidate metadata can become stale if it names a pre-merge revision or if documentation changes after artifact hashing.
- Trusted Publishing misconfiguration can authorize the wrong repository, workflow, environment, or package.
decisions:
- Keep WORK-008 preparatory and non-publishing; external release actions require a new explicit authority checkpoint.
- Use CycloneDX JSON generated by the official CycloneDX Python tool from a fresh environment containing the exact wheel and resolved runtime dependencies.
- Keep ordinary CI at contents read; isolate any future attestation permissions in a manual release-candidate workflow unavailable to pull_request events.
- Prefer PyPI Trusted Publishing with a protected GitHub environment over a long-lived API token, but do not configure either during this work item.
- Treat the merge commit after WORK-008 review as the eventual release source; pre-merge bundles are verification candidates, not publishable final artifacts.
verification:
- Begin with failing candidate-manifest, release-note, SBOM, workflow-permission, and publication-boundary tests.
- Generate the SBOM in a disposable virtual environment and validate its schema and dependency inventory with the official implementation.
- Build and verify the complete candidate bundle from a clean revision and exercise installed console commands outside the checkout.
- Run the full Critical gate locally and require GitHub CI before requesting any release authority.
next_action: Define failing candidate bundle, SBOM, release-note, and non-publishing workflow contracts before changing release implementation.
security_considerations:
- Never store PyPI tokens, GitHub tokens, OIDC tokens, signing keys, credentials, user paths, or raw environment inventories in repository artifacts.
- Attestation permissions must be absent from pull_request and ordinary CI paths; generated subjects must be selected by exact digest.
- Validate tool versions, action SHAs, archive paths, SBOM contents, hashes, and containment before accepting evidence.
- Checksums, SBOMs, and unsigned local manifests do not prove publisher identity.
recovery:
- Discard only disposable candidate directories and virtual environments on failure.
- Keep main, the verified 0.1.0 implementation, and the existing non-publishing builder as the last known-good state.
- If candidate metadata or SBOM identity drifts, rebuild everything from the reviewed source revision rather than editing generated output.
- Do not tag or publish until the exact merged release commit passes a fresh gate and the user explicitly authorizes named external actions.

# Implementation plan

1. Specify candidate identity, release-note, CycloneDX SBOM, and permission boundaries with tests.
2. Extend the non-publishing release bundle to generate and verify source-bound candidate metadata, release notes, and SBOM inputs without embedding secrets or mutable state.
3. Add a manual candidate/provenance workflow and Trusted Publishing runbook that cannot publish or run from pull requests.
4. Build and validate a disposable 0.1.0 candidate, run dependency/security checks, and reconcile remaining external prerequisites.
5. Record Critical evidence, push for review, and stop before tag, attestation, GitHub Release, environment configuration, or PyPI upload.
