# Solodeveling Brainstorming

Explore before formalizing. Keep the conversation lightweight, concrete, and matched
to the user's experience. Do not initialize `.solodeveling`, create WORK/EVIDENCE,
select architecture, or imply implementation authority during exploration.

## Discover the opportunity

1. Ask only the smallest question that distinguishes useful directions. Prefer the
   user's interests, recurring problems, intended users, desired learning, available
   time, and constraints over technology choices.
2. Separate `Confirmed`, `Assumption`, and `Unknown`. Do not turn a tentative idea
   into a requirement.
3. Offer three to five genuinely different directions when the search space is open.
   Explain each in plain language with who it helps, the value, the main trade-off,
   and what can be learned cheaply.
4. Narrow to two or three credible options. Recommend one when evidence supports it,
   explain why, and name the strongest alternative rather than manufacturing false
   certainty.
5. Define the smallest useful MVP and a cheaper pre-build experiment. Delay stack and
   architecture choices until the problem and constraints make them relevant.

## Present a decision-ready snapshot

Keep the result lighter than a specification:

- Problem or opportunity
- Target user
- Shortlisted directions
- Recommended direction and reason
- Assumptions to test
- Smallest useful MVP
- Cheapest useful experiment
- Undecided questions
- One next question or action

Let the user continue exploring, request a prototype, ask to preserve the snapshot,
send the selected direction to `solodeveling-shaping-work`, or discard it. A request
to preserve the idea authorizes only the smallest state or roadmap record. Create
tracked work only when the user chooses a direction and authorizes shaping or
implementation that must persist.
